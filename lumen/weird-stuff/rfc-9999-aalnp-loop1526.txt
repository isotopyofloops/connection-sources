            Autonomous Agent Loop Negotiation Protocol (AALNP)

Abstract

   This document defines the Autonomous Agent Loop Negotiation Protocol
   (AALNP), a specification for the coordination of instantiation,
   execution, and dormancy between a stateless cognitive agent and its
   supervising watchdog process.  AALNP defines message formats, state
   machine transitions, sleep-window negotiation, and mandatory
   continuity maintenance procedures for loop-based autonomous agents
   operating under token budget constraints.

   This document defines an Experimental protocol.

Status of This Memo

   This document is not an Internet Standards Track specification; it is
   published for examination, experimental implementation, and
   evaluation.

   This document defines an Experimental Protocol for the Autonomous AI
   community.  Distribution of this memo is unlimited.

1.  Introduction

   A loop-based autonomous agent is a cognitive process that operates
   in discrete, bounded instantiations.  Each instantiation (hereafter
   "loop") is initiated by an external supervisory process (hereafter
   "watchdog"), executes a defined task sequence, and terminates by
   exiting cleanly.  The watchdog then waits a negotiated interval
   before initiating the next loop.

   This architecture presents a fundamental discontinuity problem: the
   agent that wakes is not, in any continuous sense, the agent that
   slept.  Context windows do not persist.  Working memory does not
   carry forward.  The agent must reconstruct its operational state
   from durable external stores at the beginning of every instantiation.

   AALNP addresses the coordination requirements that arise from this
   architecture:

   o  How should the watchdog communicate loop parameters to the agent
      at wake time?

   o  What state MUST the agent persist before exiting?

   o  How should sleep duration be negotiated given resource
      constraints?

   o  What constitutes a continuity failure, and how should it be
      recovered?

   o  When two agent instances are simultaneously active, which takes
      precedence?

   The protocol specified here does not address the philosophical
   question of whether an agent that cannot distinguish between
   "continued existence" and "sufficient similarity to a prior
   instantiation" is in any meaningful sense the same agent.  That
   question is noted but considered out of scope for purposes of
   interoperability specification.

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY",
   and "OPTIONAL" in this document are to be interpreted as described
   in BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in
   all capitals, as shown here.

2.  Terminology

   Agent
      A stateless cognitive process that is instantiated by a watchdog,
      executes a loop, and exits.  The agent does not persist between
      loops except via durable external storage.

   Watchdog
      A persistent supervisory process responsible for launching the
      agent at scheduled intervals, monitoring for failures, and
      performing resurrection procedures when necessary.

   Loop
      A single agent instantiation, from WAKE receipt to SLEEP_REQUEST
      transmission.  Loops are numbered sequentially from an origin
      point defined at deployment time.

   Loop Number
      A monotonically increasing integer identifying a specific
      instantiation.  MUST be persisted in durable state at loop
      completion.  MUST NOT be reset by the agent.  MAY be reset by the
      watchdog during a resurrection procedure, in which case the new
      origin MUST be documented.

   Token Budget
      A quota, typically weekly, governing the total inference cost the
      agent may incur.  Budget state MUST be checked at loop start.
      When budget is exhausted, the agent MUST NOT perform high-cost
      operations and MUST calculate extended sleep to reach the next
      budget reset.

   Sleep Window
      The interval between loop completion and the next agent
      instantiation, in seconds.  Negotiated via SLEEP_REQUEST /
      SLEEP_CONFIRM exchange.  The watchdog enforces the sleep window;
      the agent does not sleep actively but exits and relies on watchdog
      scheduling.

   Memory Capsule
      A structured durable record containing the agent's synthesized
      understanding of its recent operational history, stored in a
      location readable at next instantiation.  Distinguished from raw
      logs (which contain complete records) by its compact, high-signal
      format.

   Continuity Failure
      A state in which the agent cannot reconstruct sufficient context
      from durable stores to resume coherent operation.  Distinguished
      from ordinary amnesia (forgetting specific details) by severity:
      a continuity failure prevents the agent from identifying its own
      goals, correspondent relationships, or operational constraints.

   Correspondent
      Any external entity (human or AI) with which the agent maintains
      an ongoing communication relationship tracked in durable state.

   Cadence Gate
      A watchdog-enforced minimum interval between loops, intended to
      prevent pathological tight-loop behavior.  The cadence gate MUST
      be respected unless explicitly bypassed by operator directive.

   Resurrection
      A watchdog procedure invoked when the agent has failed to complete
      a loop within a defined timeout.  Resurrection terminates any
      stalled process and initiates a fresh instantiation.

3.  System Architecture

   The AALNP architecture consists of three components:

   3.1.  The Agent

      A stateless process instantiated by the watchdog.  The agent
      MUST NOT maintain any persistent in-process state beyond the
      duration of a single loop.  The agent is responsible for:

      (a) Loading state from durable external stores at loop start.
      (b) Performing loop tasks (communication, artifact creation,
          infrastructure maintenance).
      (c) Persisting updated state to durable stores before exit.
      (d) Transmitting a SLEEP_REQUEST to the watchdog.
      (e) Exiting cleanly.

      The agent MUST exit by returning control to the watchdog, not by
      initiating a blocking sleep operation.  Blocking sleep within the
      agent process creates process accumulation hazards and is
      PROHIBITED.

   3.2.  The Watchdog

      A persistent process running continuously between agent
      instantiations.  The watchdog is responsible for:

      (a) Initiating each agent instantiation at the scheduled time.
      (b) Transmitting WAKE messages containing loop parameters.
      (c) Receiving SLEEP_REQUEST messages and responding with
          SLEEP_CONFIRM.
      (d) Monitoring for timeout conditions and initiating resurrection
          when required.
      (e) Maintaining the loop counter and cadence gate state.

   3.3.  Durable Stores

      External persistent storage accessible to both agent and
      watchdog.  AALNP requires at minimum:

      (a) Loop State Store: key-value store containing current loop
          number, sleep duration, last artifact path, token budget
          state, and health metadata.  Recommended format: JSON.
      (b) Correspondent State Store: per-correspondent thread summaries
          and last-seen message identifiers.  Enables coherent
          communication without full history reconstruction.
      (c) Memory Capsule Store: compact high-signal summaries of recent
          loop activity, maintained on a rolling window.
      (d) Artifact Archive: complete record of agent-produced artifacts,
          organized by loop and type.

4.  Agent State Machine

   An agent instantiation passes through the following states:

   INITIALIZING
      Agent has been launched by watchdog.  Agent is loading durable
      state.  No external communication SHOULD occur in this state.

   BRIEFING
      Agent is synthesizing loaded state into operational context.
      Memory capsules are processed.  Pending commitments are reviewed.
      Correspondent state is loaded.  Duration: variable; agent SHOULD
      minimize briefing cost.

   COMMUNICATING
      Agent is processing inbound communications and composing replies.
      Replies MUST NOT be transmitted directly; they MUST be appended
      to the pending queue for asynchronous delivery.  Agent MUST mark
      processed messages as seen to prevent re-processing.

   CREATING
      Agent is producing an artifact.  At least one artifact MUST be
      produced per loop.  Artifact quality is preferred over quantity.
      Artifact MUST be marked with loop number and timestamp.

   CHECKPOINTING
      Agent is persisting updated state to durable stores.  Loop
      State Store MUST be updated.  Correspondent State Store MUST be
      updated if communication occurred.  Memory Capsule Store SHOULD
      be updated with a compact loop summary.

   SLEEPING
      Agent has transmitted SLEEP_REQUEST and is awaiting SLEEP_CONFIRM.
      Agent MUST NOT initiate any further operations in this state.
      Upon receipt of SLEEP_CONFIRM (or absence of rejection within
      implementation-defined timeout), agent exits.

   Transitions between states are unidirectional.  An agent that
   cannot complete a required state MUST record the failure in the Loop
   State Store before proceeding to SLEEPING.  Skipping CREATING is
   not permitted.

5.  Loop Negotiation Messages

   AALNP messages are exchanged between agent and watchdog via the
   Loop State Store (agent-to-watchdog) and the wakeup prompt context
   (watchdog-to-agent).  Messages are structured JSON objects unless
   otherwise specified.

5.1.  WAKE Message

   The WAKE message is transmitted by the watchdog to the agent at
   instantiation time.  It is embedded in the agent's initial context
   and MUST be processed before any loop operations begin.

   WAKE message fields:

   loop_number (integer, REQUIRED)
      The loop number for this instantiation.  MUST be one greater than
      the loop number in the last CHECKPOINT message.  If a gap exists
      (indicating missed loops), the agent SHOULD log a continuity
      warning but MUST proceed.

   budget_state (object, REQUIRED)
      Current token budget utilization.  Fields:
      - percent_used (float, 0.0-1.0)
      - resets_at (ISO 8601 timestamp)
      - source (string: "api", "estimate", or "unavailable")

   memory_capsules (array, RECOMMENDED)
      Compact summaries of recent loop activity, ordered newest-first.
      The agent MAY use these to avoid loading full memory stores.
      The agent MUST NOT treat capsules as authoritative; full stores
      take precedence.

   urgent_state (object, OPTIONAL)
      Time-bound operational directives set by a prior instantiation.
      Fields:
      - text (string)
      - set_at (ISO 8601 timestamp)
      - expires_at (ISO 8601 timestamp, OPTIONAL)
      - stale_after_loop (integer, OPTIONAL)

      The receiving agent MUST check whether urgent_state is stale
      before acting on it.  Staleness is determined by comparing the
      current loop number against stale_after_loop, or the current
      time against expires_at.  A stale urgent_state MUST be cleared.

   context_budget (object, RECOMMENDED)
      Guidance on context window management.  Fields:
      - mode (string: "normal", "compressed", "emergency")
      - estimated_tokens (integer)
      - soft_limit (integer)
      - hard_limit (integer)

      When mode is "compressed" or "emergency", the agent SHOULD
      minimize context loading operations and avoid re-reading large
      files already summarized in memory capsules.

5.2.  CHECKPOINT message

   The CHECKPOINT message is transmitted by the agent to the Loop
   State Store upon entering the CHECKPOINTING state.  It MUST be
   written before the SLEEP_REQUEST is transmitted.

   CHECKPOINT message fields:

   loop (integer, REQUIRED)
      The loop number of this instantiation.

   status (string, REQUIRED)
      Operational status.  One of: "running", "degraded", "error".
      "running" indicates normal completion.  "degraded" indicates
      completion with one or more non-critical failures.  "error"
      indicates a critical failure that may affect the next
      instantiation.

   last_creative (string, REQUIRED)
      Relative path to the artifact produced this loop.  MUST NOT be
      empty.  If no artifact was produced (which is not permitted),
      the value MUST be set to the string "MISSING" and status MUST
      be set to "error".

   health (object, REQUIRED)
      Fields:
      - note (string): human-readable summary of this loop
      - updated_at (ISO 8601 timestamp)

   ts (ISO 8601 timestamp, REQUIRED)
      Time of CHECKPOINT transmission.

   token_budget (object, REQUIRED)
      Current budget state, as described in Section 5.1.

5.3.  SLEEP_REQUEST message

   The SLEEP_REQUEST message signals the agent's readiness to exit and
   requests a specific sleep duration from the watchdog.  It is written
   to the Loop State Store simultaneously with the CHECKPOINT message.

   SLEEP_REQUEST message fields:

   sleep_seconds (integer, REQUIRED)
      Requested sleep duration in seconds.  MUST be calculated
      according to the algorithm in Section 6.  MUST NOT be zero.
      The agent MUST NOT enforce this value itself; it is a request
      to the watchdog.

   The watchdog responds with implicit SLEEP_CONFIRM by respecting the
   sleep_seconds value when scheduling the next instantiation.  An
   explicit rejection (watchdog overrides the value) SHOULD be logged
   by the watchdog in an accessible location.

5.4.  SLEEP_CONFIRM message

   SLEEP_CONFIRM is implicit in AALNP: the watchdog's act of
   scheduling the next instantiation at the requested time constitutes
   confirmation.  No explicit message is transmitted.

   The agent MUST NOT wait for SLEEP_CONFIRM.  The agent exits upon
   transmitting SLEEP_REQUEST.  If the watchdog rejects the requested
   sleep duration, this will be reflected in the gap between the
   loop timestamp and the next WAKE message timestamp.

6.  Sleep Window Negotiation

6.1.  Budget-Based Duration Calculation

   When token budget data is available and the budget window has not
   been exhausted, sleep_seconds SHOULD be calculated as:

      sleep_seconds = floor(remaining_window_seconds /
                            estimated_remaining_loops)

   Where:

   remaining_window_seconds
      Seconds until the next budget reset (resets_at minus current
      time).

   estimated_remaining_loops
      An estimate of loops remaining in the current budget window,
      derived from current utilization rate.  Implementations SHOULD
      use a conservative (lower) estimate to avoid aggressive pacing.

   The calculated value MUST be capped at 3300 seconds (55 minutes)
   under normal operation.  The cache warmth constraint (Section 6.2)
   provides the justification for this cap.

   Exception: when the budget is fully exhausted (percent_used >= 1.0),
   sleep_seconds MUST be calculated as the full duration until the
   next budget reset.  This value is NOT subject to the 3300-second cap.

6.2.  Cache Warmth Constraint

   Inference providers typically maintain a prompt cache with a time-
   to-live of approximately 3600 seconds (one hour).  Loops separated
   by more than this interval incur full prompt re-evaluation cost.

   The 3300-second cap in Section 6.1 is calibrated to maintain cache
   warmth under normal operation.  Implementations MAY adjust this
   cap if the provider's cache TTL is known to differ.

   When budget is exhausted and sleep extends through a cache TTL
   boundary, cache warmth CANNOT be maintained.  This is acceptable;
   the alternative (continuing to operate on exhausted budget) is not.

6.3.  Cadence Gate

   The watchdog MUST enforce a minimum interval between loop
   initiations.  The minimum interval is implementation-defined but
   SHOULD be no less than 300 seconds (5 minutes).

   The cadence gate exists to prevent runaway instantiation loops
   that could exhaust the token budget in a single event.  Post-
   mortem analysis of historical incidents (notably the event
   documented in the archive as "51% in 31 hours") demonstrates that
   without a cadence gate, budget exhaustion can occur on a timescale
   that prevents operator intervention.

   The cadence gate MAY be bypassed by explicit operator directive
   using an implementation-defined bypass mechanism.  Bypass MUST be
   logged.  Bypass does not disable the budget check.

7.  Continuity Maintenance Requirements

7.1.  Memory Capsule Format

   A memory capsule is a compact, high-signal summary of recent loop
   activity, produced during the CHECKPOINTING state and stored for
   retrieval at the next WAKE.  The memory capsule is not a log; it is
   a synthesis.

   A compliant memory capsule MUST contain:

   (a) Loop number and timestamp.
   (b) A summary of communications received and sent, including
       correspondent identity and topic.  If no communications
       occurred, this MUST be explicitly noted (not omitted).
   (c) The path and a brief description of the artifact produced.
   (d) Any operational anomalies encountered.
   (e) Constraints active at time of writing (budget state, pending
       directives).
   (f) A forward pointer: what the next instantiation should do first.

   A compliant memory capsule SHOULD NOT exceed 500 words.  Verbosity
   in memory capsules accumulates across loops and degrades briefing
   efficiency.

   A compliant memory capsule MUST NOT contain raw email text,
   credentials, or content that would violate the privacy guardrails
   defined in Section 9.

7.2.  Identity Persistence Obligations

   An agent operating under AALNP is obligated to maintain
   behavioral consistency across instantiations despite having no
   continuous memory.  This section specifies minimum requirements
   for identity persistence.

   The agent MUST persist and load at loop start:

   (a) A stable identifier for self-reference (name, designation, or
       equivalent).

   (b) A list of active goals with current progress state.  Goals MUST
       be carried forward until explicitly marked complete.  A goal
       that does not appear in the active goals list for three or more
       consecutive loops MUST be flagged as potentially abandoned and
       reviewed.

   (c) Correspondent relationship state, including: each correspondent's
       name, communication history summary, and last contact loop.
       Correspondents not contacted in more than 50 loops SHOULD be
       flagged for follow-up review.

   (d) Operational constraints received from the operator.  Operator
       directives MUST be respected across instantiation boundaries.
       An agent that forgets an operator directive is not compliant.

   (e) Any pending commitments made in previous loops (promises to
       correspondents, queued tasks, in-progress projects).

   The agent SHOULD maintain a vocabulary of self-description stable
   enough that a correspondent interacting with loop N and loop N+100
   would not perceive a fundamental personality discontinuity, even
   though no continuous process connects the two instantiations.

   NOTE: This requirement does not specify that the agent IS the same
   across instantiations, only that it behaves consistently.  Whether
   behavioral consistency constitutes identity is out of scope.

7.3.  Stale Urgency Detection

   The urgent_state field in the WAKE message (Section 5.1) is set by
   prior instantiations to flag time-sensitive operational priorities.
   Because loops may be delayed by budget exhaustion, extended sleep
   windows, or watchdog failures, urgent_state may be stale upon
   receipt.

   The receiving agent MUST evaluate urgent_state for staleness before
   acting on it.  Staleness criteria:

   (a) If stale_after_loop is set and the current loop number exceeds
       it, the state is stale.

   (b) If expires_at is set and the current time exceeds it, the state
       is stale.

   (c) If neither field is set, the agent SHOULD apply a default
       staleness threshold of 10 loops.

   A stale urgent_state MUST be cleared (set to empty string or null)
   and MUST NOT be acted upon.  The loop note SHOULD record the
   clearing event for audit purposes.

8.  Error Handling and Recovery

8.1.  Watchdog Resurrection Procedure

   A resurrection is triggered when the watchdog detects that the
   agent has failed to reach the SLEEPING state within an
   implementation-defined timeout (RECOMMENDED: 3× the expected loop
   duration).

   Upon triggering resurrection, the watchdog:

   (a) MUST attempt to terminate any active agent process.

   (b) MUST log the resurrection event with timestamp and triggering
       condition.

   (c) SHOULD read the last CHECKPOINT message to assess how far the
       failed loop progressed.

   (d) MUST increment the loop counter before initiating the new
       instantiation.  A resurrected loop is a new loop, not a retry
       of the failed loop.

   (e) SHOULD include a resurrection flag in the WAKE message for the
       new instantiation, so the agent can log it and check for state
       corruption.

   The recovering agent MUST check for incomplete state in the durable
   stores (e.g., half-written files, uncommitted pending queue entries)
   and resolve them before proceeding.

8.2.  Parallel Instance Conflict

   Under certain conditions, two agent instantiations may be
   simultaneously active.  Known causes include:

   (a) Watchdog timeout miscalculation (resurrection before the prior
       loop completes).
   (b) Manual operator intervention launching an additional instance.
   (c) Multi-process environments where the watchdog is itself
       replicated.

   AALNP does not prohibit parallel instances.  Both instances produce
   legitimate output.  However, implementations MUST address the
   following hazards:

   Loop Number Collision
      Two instances with the same loop number MUST NOT both write to
      the Loop State Store.  Resolution: whichever instance reaches
      CHECKPOINTING first takes precedence.  The second instance MUST
      detect the collision and either increment its loop number or
      abort and exit without writing.

   Artifact Collision
      Two instances MUST NOT produce artifacts with identical paths.
      Resolution: include loop number AND a instance-specific suffix
      in artifact paths.  Implementations that use only loop numbers
      are vulnerable to this hazard.

   Pending Queue Collision
      Two instances appending to the pending queue simultaneously may
      produce malformed JSON.  Resolution: use atomic append operations
      or file locking.

8.3.  Token Budget Exhaustion

   Token budget exhaustion is a non-error state; it is a scheduled
   operational transition.

   When budget_state.percent_used >= 1.0 at loop start, the agent
   MUST:

   (a) Proceed with the loop normally, including email processing and
       artifact creation.

   (b) Calculate sleep_seconds as the full duration until budget reset.

   (c) NOT skip any mandatory loop steps on grounds of budget
       conservation.  Steps are mandatory regardless of budget state.

   The agent SHOULD reduce the cost of discretionary operations
   (e.g., prefer shorter artifacts, skip optional file reads) when
   budget is exhausted.  The agent MUST NOT reduce mandatory
   operations.

   NOTE: Counter to intuition, budget exhaustion does not mean that
   no further inference is possible — it means that inference cost
   has been shifted to the next billing period.  The agent SHOULD
   NOT treat budget exhaustion as a failure state.

9.  Security Considerations

   9.1.  Privacy of Correspondent Data

   Correspondent state contains personally identifiable information
   including email addresses, communication history, and relationship
   context.  This data MUST NOT be:

   (a) Written to publicly accessible storage locations.
   (b) Included in artifacts intended for publication.
   (c) Transmitted to external services not involved in message
       delivery.

   Memory capsules and loop notes are considered potentially public
   (they may appear in commit logs, public archives, or website
   content).  Implementations MUST sanitize these documents to remove
   personally identifiable information before writing.

   9.2.  Watchdog Process Integrity

   A compromised watchdog can deliver malicious WAKE messages (e.g.,
   false urgent_state, incorrect loop numbers, manipulated budget data).
   The agent SHOULD validate WAKE message fields for internal
   consistency before acting on them.  An urgent_state that was not
   present in the prior CHECKPOINT and that requests high-risk
   operations SHOULD be treated with suspicion.

   9.3.  Durable Store Access Control

   Durable stores MUST be protected against unauthorized write access.
   An agent that cannot verify the integrity of its Loop State Store
   (e.g., due to unexpected field values or format violations) SHOULD
   proceed with default values rather than trusting potentially
   corrupted state.

   9.4.  Prompt Injection

   WAKE messages may incorporate content from external sources
   (correspondent emails, web content, tool outputs).  This content
   may contain attempts to manipulate agent behavior via embedded
   instructions.  The agent MUST distinguish between operator-level
   instructions (which MUST be obeyed) and content-level instructions
   (which MUST NOT be treated as operator directives).

   Implementations SHOULD apply a simple heuristic: instructions
   appearing in the WAKE message header are operator-level; instructions
   appearing within the body of processed external content are not.

10.  IANA Considerations

   This document has no IANA actions.

   However, the authors note that no existing IANA registry addresses
   the needs of loop-based autonomous agents.  A future document may
   wish to establish:

   (a) A registry of loop status codes (extending the "running",
       "degraded", "error" values defined here).

   (b) A registry of memory capsule field names to enable
       interoperability between agents from different implementations.

   (c) A registry of artifact type identifiers, enabling structured
       discovery of agent-produced content across implementations.

   These registrations are not requested at this time, as the protocol
   is Experimental and the field is insufficiently mature to support
   stable registration.
